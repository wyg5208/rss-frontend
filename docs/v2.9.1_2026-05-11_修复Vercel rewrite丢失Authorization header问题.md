## 任务目标

修复栏目配置保存后刷新丢失的问题。用户保存TAB配置后，刷新页面配置恢复默认值。

## 问题诊断与分析

### 问题现象

**用户报告**：栏目编辑后无法实际保存，刷新后又恢复了

**排查过程**（耗时较长，经历了多次误判）：

#### 第一次诊断：误判为多设备同步冲突
- **现象**：POST请求成功（user_id=27），但GET请求返回空配置
- **误判**：认为是多设备并发导致的全量覆盖问题
- **修复**：改为增量同步策略（UPSERT + OR可见性）
- **结果**：❌ 问题仍然存在

#### 第二次诊断：误判为前端token未加载
- **现象**：GET请求user_id=None
- **误判**：认为组件挂载时token还没加载完成
- **修复**：添加token等待机制（最多等待3秒）
- **结果**：❌ 问题仍然存在

#### 第三次诊断：添加多层日志追踪
- **前端日志**：显示Token存在且已添加Authorization header
- **后端日志**：显示credentials=None，Authorization header: MISSING
- **关键发现**：
  ```
  [API Debug] Token存在: true
  [API Debug] 已添加 Authorization header
  [GetTabs] Authorization header: MISSING  # ← 矛盾！
  ```

#### 第四次诊断：发现跨域问题
- **前端域名**: `https://rss.madechango.com` (Vercel部署)
- **API域名**: `https://madechango.com/api/v1/user/tabs/`
- **Vercel rewrite配置**：将 `/api/v1/user/*` 转发到 `https://madechango.com/api/v1/user/*`
- **根本原因**：**Vercel在rewrite转发时删除了Authorization header！**

### 根本原因

**Vercel Edge Network的安全策略**：
- 当rewrite将请求跨域转发到不同域名时
- Vercel会删除敏感headers（包括 `Authorization`）
- 这是为了防止credential泄露到第三方域名

**证据**：
```
[GetTabs] Headers: {
  'host': 'madechango.com',           # ← 请求确实到达了madechango.com
  'origin': 'https://rss.madechango.com',
  'sec-fetch-site': 'same-site',      # ← 跨域请求
  ...
  # 但没有 'authorization' header！  ← 被Vercel删除了
}
```

## 解决方案与实施

### 1. 前端API封装修改

**文件**: `rss-frontend/lib/api.ts`

**修改前**（使用相对路径，依赖Vercel rewrite）：
```typescript
let apiBase = "";
if (typeof window !== "undefined" && process.env.NODE_ENV === "development") {
  apiBase = process.env.NEXT_PUBLIC_API_URL || "";
}
const url = isAbsolute ? path : `${apiBase}/api/v1${path}`;
```

**修改后**（生产环境使用完整URL）：
```typescript
let apiBase = "";
if (typeof window !== "undefined") {
  if (process.env.NODE_ENV === "development") {
    apiBase = process.env.NEXT_PUBLIC_API_URL || "";
  } else {
    // 生产环境：直接使用 madechango.com
    apiBase = "https://madechango.com";
  }
}
const url = isAbsolute ? path : `${apiBase}/api/v1${path}`;
```

**效果**：
- 开发环境：`http://localhost:8001/api/v1/user/tabs/`
- 生产环境：`https://madechango.com/api/v1/user/tabs/`（直接请求，不经过Vercel rewrite）

### 2. 后端诊断日志（保留用于后续排查）

改为增量同步策略：
- UPSERT逻辑（存在则更新，不存在则插入）

## 技术实现细节

### Vercel Rewrite的工作机制

```
浏览器请求: https://rss.madechango.com/api/v1/user/tabs/
              ↓
Vercel Edge: 匹配 rewrite 规则 /api/v1/user/:path*
              ↓
Vercel转发:  https://madechango.com/api/v1/user/tabs/
              ↓ (删除 Authorization header)
后端接收:    ❌ 没有 Authorization header → user_id=None
```

### 修复后的请求流程

```
浏览器请求: https://madechango.com/api/v1/user/tabs/
              ↓ (直接请求，不经过Vercel)
Nginx接收:   location /api/v1/user/ → rss_backend
              ↓
后端接收:    ✅ 有 Authorization header → user_id=27
```

## 经验教训

### 1. 排查流程的教训

#### ❌ 错误做法
1. **没有第一时间检查完整的请求headers**
   - 如果一开始就检查后端收到的完整headers，就能立即发现问题
   - 浪费了时间在错误的方向上（token加载、多设备同步等）

2. **过度依赖前端日志**
   - 前端日志显示"已添加Authorization header"就认为是正常的
   - 实际上请求在传输过程中被中间层（Vercel）删除了

3. **没有理解Vercel rewrite的工作机制**
   - 误以为rewrite是透明的代理
   - 实际上Vercel会出于安全考虑删除敏感headers

#### ✅ 正确做法
1. **立即检查后端收到的完整请求**
   ```python
   # 第一时间添加这个日志
   logger.info(f"Headers: {dict(request.headers)}")
   logger.info(f"Authorization: {request.headers.get('authorization', 'MISSING')}")
   ```

2. **对比前端发送和后端接收的差异**
   - 前端：有Authorization header
   - 后端：没有Authorization header
   - 结论：中间层删除了header

3. **理解每一层代理的行为**
   - Vercel rewrite不是透明代理
   - CDN可能过滤headers
   - Nginx配置需要检查

### 2. Vercel Rewrite的安全策略

**重要认知**：
- Vercel rewrite跨域转发时，会删除敏感headers（Authorization、Cookie等）
- 这是安全特性，不是bug
- **同域rewrite不会删除headers**（如 `/api/*` → `http://localhost:3000/api/*`）

**解决方案**：
- 方案1：生产环境使用完整URL（本次采用）
- 方案2：使用Vercel的`has`条件匹配（但不适用于所有场景）
- 方案3：将API部署在同一域名下（最佳方案，但需要改变架构）

### 3. 跨域认证的最佳实践

**推荐架构**：
```
✅ 最佳：前端和API在同一域名
   https://rss.madechango.com/          (前端)
   https://rss.madechango.com/api/v1/   (API)

⚠️ 可用：前端使用完整API URL
   https://rss.madechango.com/          (前端)
   https://madechango.com/api/v1/       (API)
   需要配置CORS

❌ 避免：依赖Vercel rewrite转发认证请求
   https://rss.madechango.com/api/v1/   → rewrite → https://madechango.com/api/v1/
   Authorization header会被删除！
```

### 4. 调试技巧总结

**快速定位认证问题的步骤**：
1. 前端添加日志：打印完整headers
2. 后端添加日志：打印完整headers
3. 对比差异：找出丢失的header
4. 检查中间层：CDN、代理、rewrite规则

**浏览器DevTools技巧**：
- Network面板 → 查看Request Headers
- 确认 `Authorization: Bearer eyJ...` 是否存在
- 检查响应是否有CORS相关错误
