## 问题诊断与分析

### 问题 1: 大表ALTER TABLE超时

**现象**: 在12万行的rss_articles表上添加翻译字段时，ALTER TABLE操作超时（30秒）

**原因**: MySQL在超大型表上修改表结构需要重构表，时间超过30秒默认超时

**解决方案**: 采用独立翻译表设计，创建article_translations表关联主表

### 问题 2: 数据库连接泄漏导致服务卡死

**现象**: 服务运行一段时间后卡死，无法响应HTTP请求

**原因**: get_db_context_for_thread()每次创建新引擎但没有dispose()释放连接

**解决方案**: 添加finally块确保engine.dispose()释放连接

### 问题 3: 翻译服务连接管理不当

**现象**: 每次翻译创建多个数据库连接，性能低下

**原因**: _get_article()和_save_translation()分别创建独立的数据库上下文

**解决方案**: 合并到单个数据库上下文，减少5倍连接创建

**database.py**
```python
@asynccontextmanager
async def get_db_context_for_thread():
    engine = create_async_engine_for_thread()
    # ...
    try:
        yield session
        await session.commit()
    except:
        await session.rollback()
        raise
    finally:
        # 关键：释放引擎和连接池
        await engine.dispose()
```

## 经验教训

### 1. 大表结构变更策略

在大表（>10万行）上添加字段时，应该：
- 优先考虑独立表设计
- 避免直接ALTER TABLE
- 使用外键关联保持数据一致性

### 2. 数据库连接管理

asynccontextmanager必须确保：
- try-finally结构
- engine.dispose()在finally中
- 异常情况也能正确释放资源

### 3. 前端用户体验

翻译功能应该：
- 无标签，简洁显示
- 淡入动画提升体验
- 状态反馈及时准确
