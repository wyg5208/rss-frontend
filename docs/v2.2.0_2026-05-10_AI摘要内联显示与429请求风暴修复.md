## 问题诊断与分析

### 问题 1: 前端429请求风暴
**现象**: 前端一次性发送90+个API请求（skip=0到skip=1860），导致后端429 Too Many Requests
**原因**: 两层级联触发：
1. `useArticles.ts` 中有 `useEffect` 自动调用 `fetchNextPage()`，每次加载完一页就触发下一页
2. `ArticleList.tsx` 的 `IntersectionObserver` 在加载新数据后，最后一个元素可能立刻可见，又触发 `fetchNextPage()`
3. 两者叠加形成请求风暴

### 问题 2: AI摘要首次生成后无法显示
**现象**: 点击AI摘要按钮生成后，摘要区域一直空白；但如果文章已有摘要，点击后能正常显示
**原因**: `POST /ai/process/{id}` 只返回状态（`summary_success: true`），不返回实际内容（`ai_summary` 为 undefined），前端从 `response.data.ai_summary` 取值永远为空

### 问题 3: AI摘要弹窗与双语翻译体验不统一
**现象**: AI摘要使用模态弹窗显示，而双语翻译使用内联显示，体验不一致
**原因**: FloatingToolbar 内部维护了 `aiModalOpen` 状态和弹窗UI，与双语翻译的内联模式不对齐

### 问题 4: 悬浮工具栏遮挡阅读
**现象**: 工具栏背景不够透明，影响阅读体验
**原因**: 透明度从85%→95%→15%→2%逐步调整，最终确认完全透明方案

### 问题 5: 黑名单操作后跳转不合理
**现象**: 加入不看后3秒返回首页，体验不连贯
**原因**: 硬编码 `router.push("/")` 返回首页，没有考虑上下文

## 解决方案与实施

### 1. 429请求风暴修复（双重防护）

**防护层1 - useArticles.ts**:
- 移除自动预取的 `useEffect`
- 添加 `MAX_PAGES = 10` 安全限制（最多加载200条）
- `retry: 1` 限制重试次数，避免429雪崩
- `retryDelay: 2000` 设置重试间隔

**防护层2 - ArticleList.tsx**:
- 添加 `fetchingRef` 防重复触发守卫
- 1秒冷却期，防止加载状态还未更新就再次触发
- `rootMargin` 从 100px 改为 200px

### 2. AI摘要显示机制重构

**阶段1 - 修复首次生成不显示**:
- 改为生成成功后调用 `loadAIData()` 从 `GET /ai/article/{id}` 获取完整数据

**阶段2 - 防重复生成+直接显示**:
- 如果已有AI摘要，切换显示/隐藏而非重新生成
- 加载AI数据时，如有摘要自动设置 `aiVisible = true`

**阶段3 - 内联显示重构**:
- 新增 `AISummaryCard` 组件（内联显示，替代弹窗）
- FloatingToolbar 移除弹窗代码（减少70+行）
- AI摘要按钮改为切换显示/隐藏
- 文章详情页增加 `aiVisible` 状态控制

### 3. 悬浮工具栏优化

- 背景改为 `bg-transparent` 完全透明
- 移除 `backdrop-blur` 毛玻璃效果和 `shadow-lg` 阴影
- 新增垂直拖拽功能，位置范围 10%-90%
- 垂直位置 localStorage 持久化

### 4. 黑名单跳转优化

- 优先跳转到下一篇文章（1秒后）
- 无下一篇时返回上一页
- 取消3秒等待改为1秒

### 5. 双语翻译标签优化

- TranslatedText 组件新增 `type` 属性区分标题/摘要翻译
- 标题翻译显示"译名"标签，摘要翻译显示"译文"标签
- 原文摘要独立显示（非双语模式下）

const lastElementRef = useCallback(
  (node: HTMLDivElement | null) => {
    if (loading) return;
    if (observer.current) observer.current.disconnect();
    observer.current = new IntersectionObserver((entries) => {
      if (entries[0].isIntersecting && hasMore && !fetchingRef.current) {
        fetchingRef.current = true;
        onLoadMore();
        // 1秒冷却期，防止加载状态还没更新就再次触发
        setTimeout(() => { fetchingRef.current = false; }, 1000);
      }
    }, { rootMargin: "200px" });
    if (node) observer.current.observe(node);
  },
  [loading, hasMore, onLoadMore]
);

## 经验教训

### 1. useInfiniteQuery 的 useEffect 自动预取是危险的
- `useEffect` 监听 `query.data.pages.length` 变化来触发 `fetchNextPage()` 会导致级联请求
- 应该由用户交互（滚动）触发加载，而非数据变化自动触发
- 即使移除了 useEffect，IntersectionObserver 也需要防级联守卫

### 2. IntersectionObserver 需要防重复触发
- React 状态更新是异步的，`loading` 状态不会立即变为 true
- 使用 `useRef` 作为同步守卫比依赖 React 状态更可靠
- 冷却期机制（1秒）是必要的，避免快速连续触发

### 3. API 接口返回值需要与前端消费方式匹配
- `POST /ai/process/{id}` 只返回状态不返回内容，前端不能从 response 中直接取 ai_summary
- 需要额外调用 `GET /ai/article/{id}` 获取完整数据
- 或者后端 process 接口直接返回生成的AI数据

### 4. 悬浮工具栏透明度的渐进式调整
- 85% → 95% → 15% → 2% → 完全透明，说明需要多次迭代找到合适值
- 完全透明 + 细边框 是移动端悬浮组件的最佳实践
