## 任务目标

优化 RSS 移动端文章详情页面的双语翻译功能，解决以下问题：
1. 长摘要文章翻译结果不显示的问题
2. 翻译结果格式不统一（部分带 HTML 标签，部分不带）
3. 前端 API 调用缺乏超时控制
4. 错误处理和用户体验不完善

## 问题诊断与分析

### 问题 1: 长摘要翻译结果不显示
**现象**: 当文章摘要很长时（实际是文章内容），即使 AI 生成翻译成功，前端也不显示翻译结果

**原因**: 
1. **后端 max_tokens 限制过低**：原设置 `max_tokens=500`，长摘要翻译时结果被截断
2. **JSON 解析失败**：截断后的结果 JSON 格式不完整，导致 `_parse_translation_result` 解析失败
3. **前端无超时控制**：API 调用没有超时机制，长文本翻译可能挂起
4. **状态管理缺陷**：`bilingualOn` 和 `translatedData` 状态不同步

### 问题 2: 翻译结果格式不统一
**现象**: 不同文章翻译后，有的带 HTML 标签（`<p>`, `<br>`, `<strong>` 等），有的是纯文本

**原因**:
1. **多模型输出不一致**：DeepSeek、GLM、Kimi、Qwen 对提示词遵循程度不同
2. **提示词约束不足**：未明确要求返回纯文本、移除 HTML 标签
3. **缺少后处理清理**：解析后没有对内容进行 HTML 标签过滤

### 1. 后端翻译服务优化

#### 1.1 超长摘要优雅截断（发送前处理）

**实现逻辑**:
```python
# 限制发送的摘要长度（约 3000 字符）
max_summary_length = 3000

if len(summary) > max_summary_length:
    # 优雅截断：在句子边界处截断
    truncated_summary = summary[:max_summary_length]
    # 找到最后一个完整的句子结束位置
    last_sentence_end = max(
        truncated_summary.rfind('.'),
        truncated_summary.rfind('。'),
        truncated_summary.rfind('!'),
        truncated_summary.rfind('！'),
        truncated_summary.rfind('?'),
        truncated_summary.rfind('？'),
        truncated_summary.rfind('\n')
    )
    
    if last_sentence_end > max_summary_length * 0.5:
        summary = truncated_summary[:last_sentence_end + 1] + "..."
    else:
        summary = truncated_summary + "..."
```

**优势**:
- ✅ 节省 AI 调用成本（不发送过长文本）
- ✅ 提高翻译质量（截断后更聚焦）
- ✅ 保持语义完整性（在句子边界截断）

**文件**: `d:\python_projects\rss-frontend\lib\api.ts`

```typescript
async function apiFetch<T>(path: string, options?: RequestInit & { timeout?: number }): Promise<T> {
  // 创建 AbortController 用于超时控制
  const controller = new AbortController();
  const timeout = options?.timeout || 30000; // 默认 30 秒超时
  const timeoutId = setTimeout(() => controller.abort(), timeout);

  try {
    const res = await fetch(url, { 
      ...options, 
      headers,
      signal: controller.signal 
    });
    // ...
  } finally {
    clearTimeout(timeoutId);
  }
}
```

**API 方法签名更新**:
```typescript
export const api = {
  get: <T>(path: string, options?: RequestInit & { timeout?: number }) => apiFetch<T>(path, options),
  post: <T>(path: string, body?: unknown, options?: RequestInit & { timeout?: number }) =>
    apiFetch<T>(path, {
      method: "POST",
      body: body ? JSON.stringify(body) : undefined,
      ...options,
    }),
  delete: <T>(path: string, options?: RequestInit & { timeout?: number }) => 
    apiFetch<T>(path, { method: "DELETE", ...options }),
};
```

## 经验教训

### 1. 经验总结

#### 长文本处理策略
- **发送前截断 + 足够输出空间** 是最佳实践
- 截断应优先在**语义边界**（句子结束处）进行
- 不要依赖 AI 模型自动处理超长文本

#### 防御性编程
- **不信任外部输入**：AI 模型的输出必须验证和清理
- **双重保障**：提示词约束 + 后处理清理
- **容错设计**：部分成功也应展示给用户

#### API 设计
- 所有 API 调用必须有**超时控制**
- 区分不同类型的错误，提供明确的错误提示
- 类型定义要完整，避免运行时错误

### 2. 注意事项

⚠️ **截断位置选择**
- 优先在句子边界（`。！？.\n`）处截断
- 如果前 50% 找不到句子边界，直接截断
- 截断后添加 "..." 表示内容不完整

⚠️ **max_tokens 设置**
- 输出空间必须足够，避免结果被截断
- 10000 tokens 约等于 7000-8000 个中文字符
- 根据实际业务需求调整

⚠️ **HTML 标签清理**
- 使用正则 `re.sub(r'<[^>]+>', '', text)` 移除所有标签
- 注意保留必要的空格和换行
- 标题和摘要的格式化策略不同

⚠️ **超时时间设置**
- 翻译请求比普通请求需要更长时间
- 60 秒是合理的翻译超时时间
- 超时后应释放资源并提示用户
